## Abstract

Cardano transactions follow a sequential flow: construct valid transaction (build), add witness signatures (sign), and broadcast to network (submit). The architecture enforces this ordering through types—attempting operations out of sequence produces compile-time errors. The critical design decision is **fresh state per execution**: each `build()` call creates independent state from stored program descriptions, enabling safe builder reuse without state contamination.

## Purpose and Scope

This document explains how transaction construction separates **program description** (what to do) from **program execution** (doing it), and why this separation enables safe reuse patterns. It covers the builder→sign→submit flow for signing wallets, and the builder→result flow for read-only wallets.

**Not covered**: Specific API methods for building transactions (see transaction guides), coin selection algorithms (see provider layer), or wallet signing internals (see wallet layer).

## Design Philosophy

Traditional mutable builders face a dilemma: execute operations immediately (preventing inspection before commitment) or accumulate mutable state (creating unsafe reuse patterns). Immediate execution couples program description with execution timing. Mutable state creates cross-contamination when builders are reused.

The architecture resolves this by storing **program descriptions** without executing them. When `build()` is called, the builder creates fresh state and runs all programs sequentially. The same builder can be called multiple times—each execution starts from an empty slate with no contamination from previous runs.

This separation enables template reuse (define a payment pattern once, execute multiple times) and pre-execution inspection (examine the program sequence before committing resources).

**[1] SigningTransactionBuilder**: Wallet can sign (seed phrase, private key, or CIP-30)

- Type system prevents `sign()` and `submit()` methods from existing

## Fresh State Execution Model

Each `build()` call creates completely independent state. The builder stores program descriptions permanently, but execution state lives only for one build cycle:

<Mermaid chart={`
sequenceDiagram
    participant User
    participant Builder
    participant State
    participant Provider

    User->>Builder: newTx().payToAddress(...)
    Note over Builder: [1] Store program<br/>NO execution
    
    User->>Builder: build() #1
    Builder->>State: [2] Create fresh Refs<br/>outputs: []<br/>inputs: []
    Builder->>State: [3] Execute all programs
    State->>Provider: Query UTxOs
    Provider-->>State: Return UTxOs
    State->>State: Select inputs<br/>Balance outputs
    State-->>User: Result #1
    
    User->>Builder: build() #2
    Builder->>State: [2] Create NEW fresh Refs<br/>outputs: []<br/>inputs: []
    Note over State: Completely independent
    Builder->>State: [3] Execute all programs
    State->>Provider: Query UTxOs (fresh)
    Provider-->>State: Return UTxOs
    State->>State: Select inputs<br/>Balance outputs
    State-->>User: Result #2
`} />

**[1] Program Storage**: Builder methods (payToAddress, collectFrom) append to an immutable program array. No I/O happens yet.

**[2] Fresh State Creation**: Each `build()` creates new `Ref` instances for outputs, inputs, scripts. Previous execution state is discarded.

**[3] Program Execution**: All stored programs run sequentially against fresh state. UTxO queries, coin selection, fee calculation—all independent per execution.

This model provides three critical guarantees:
1. **Reusability**: Same builder works for multiple transactions without state contamination
2. **Predictability**: Each execution starts from known empty state, no accumulated side effects
3. **Concurrency**: Multiple `build()` calls can run concurrently (on different instances) without interference

## Build Phase State Machine

The `build()` execution runs through a multi-phase state machine where each phase handles one aspect of transaction construction and returns the next phase to execute. Each phase is shown below with its complete decision logic derived from the source code.

## [1] Selection Phase

Orchestrates coin selection to cover transaction requirements, handling both initial selection and reselection when additional inputs are needed:

<Mermaid chart={`

    CheckSufficient -->|No| IsReselection{Reselection?<br/>shortfall > 0}

    IsReselection -->|No| SelectShortfall[Select for<br/>Asset Delta]
    
    ReselectLovelace --> IncrementAttempt[Increment Attempt]

    SkipSelection --> IncrementAttempt
    
    IncrementAttempt --> HasScripts{Has<br/>Redeemers?}
    HasScripts -->|Yes| ToCollateral([Next: collateral])
    HasScripts -->|No| ToChange([Next: changeCreation])

**Key Phase Behaviors:**

- **Sufficiency Check**: Skips coin selection if explicit inputs (from `collectFrom()`) already cover requirements
- **Normal Selection**: Invokes coin selection algorithm for asset delta when initial inputs insufficient
- **Reselection Mode**: When change creation sets shortfall, selects specifically for that shortfall amount
- **Script Routing**: Routes to collateral phase first when redeemers exist (scripts need collateral before change)
- **Attempt Tracking**: Increments attempt counter after each pass, reset at phase start

Routes to **collateral** first if scripts are present (before change creation), or **changeCreation** for non-script transactions. Reselection adds more UTxOs when shortfall exists from fee increases or minUTxO requirements.

**See Also**: [Coin Selection](/docs/architecture/coin-selection) for algorithm details, largest-first strategy, multi-asset handling, and custom selection functions.

**Key Phase Behaviors:**

- **Conditional Execution**: Skips entirely if no redeemers present (ADA-only transactions)
- **Loop Prevention**: Checks if all redeemers already evaluated to prevent infinite re-evaluation
- **Evaluator Resolution**: Uses provider-based evaluation (default) or custom UPLC evaluator (e.g., Aiken)
- **ExUnits Update**: Writes computed memory and CPU steps back to redeemer entries
- **Fee Impact**: Updated ExUnits change transaction size, triggering fee recalculation

Always proceeds to **feeCalculation**. Re-evaluation occurs when ExUnits are invalidated after reselection changes the input set.

**See Also**: [Script Evaluation](/docs/architecture/script-evaluation) for detailed algorithm explanation, custom evaluator patterns, and UPLC execution mechanics.

## Integration Points

The transaction builder integrates with two key layers:

**Provider Layer**: During the `build()` phase, the builder queries the provider for:
- Protocol parameters (fees, transaction size limits)
- Available UTxOs at wallet addresses
- Script evaluation (when Plutus scripts are attached)

**Wallet Layer**: The wallet type determines builder capabilities:
- **Signing wallets** (seed, private key, CIP-30) → `SigningTransactionBuilder` → full build/sign/submit flow
- **Read-only wallets** → `ReadOnlyTransactionBuilder` → build only, returns unsigned transaction
- Type system enforces these boundaries at compile time—cannot call `sign()` on read-only builder
